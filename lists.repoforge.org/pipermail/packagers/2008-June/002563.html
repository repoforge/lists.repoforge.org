On Sun, 1 Jun 2008, Quien Sabe wrote:

> i am working on a personal investments project that requires a "moving 
> average" calculator
>
> I found perl-Math-Business-SMA ( and perl-Math-Business-MACD )
>
> Investigation on CPAN found newer versions of these modules packaged together 
> with several others as the "perl-stockmonkey" package.
>
> What is the correct way to handle this in rpmforge?
>
> 1. A new package 'perl-stockmonkey' which obsoletes the older discrete 
> packages

If you are sure that these would otherwise conflict and that these are in 
fact newer versions, then yes, an obsolete would be required.


> or
>
> 2. Update the older packages to build from the stockmonkey source, and create 
> the other discrete module packages needed?

It all depends what upstream thinks about it. It is possible that the 
author of perl-stockmonkey integrated someone else's package and changed 
the version because the original was no longer updated. But that may 
become problematic if the original author restarts maintenance.

The important questions are:

  - do the locations of both really conflict ?
  - are the versions really different ?
  - who is maintaining the modules in the future ?
  - is there a chance this will be reverted ?

Contacting both authors is probably best.
