I'm to blame for everything that's not clear. I think the first thing to 
do is to acknowledge you're interested to join RPMforge and give a list of 
packages. The normal procedure would be to send 1 mail per 
package/SRPM/specfile/request to suggest at lists.rpmforge.net and then we'll 
answer with either some remarks or with an acknowledgment the package has 
been added.

If you indicate that you want to maintain the package it depends on the 
amount of package you maintain (the amount of work), the skills and trust
before you get granted access to subversion. In your case this will not be 
a problem.

If there are other things unclear (like policies around SPEC files and 
builds) just ask and we'll answer, but nothing is set in stone. I hope to 
list a FAQ eventually on the website so we don't have to answer 
packager questions multiple times. (A wiki might also be nice for this)

Eventually Dries and I will build the packages in our repositories, but 
the aim is to merge them as soon as possible and make different persons 
responsible for building for 1 platform. (Instead of duplicating the work)

Like FreshRPMS, it is still possible to create your own repository out of 
subversion and brand/customize them, but at least for the duplicate 
packages you will become the maintainer and they will be compatible with 
other repositories that are based on or compatible with RPMforge.
(That's the middleground we can offer, I prefer this over duplicated work 
even though it is still suboptimal)

Dries is finishing the first public release of pydar. pydar would make it 
possible to queue packages to build them for different platforms (on 
different servers).

There are some other things we have to discuss, like the repository setup 
for 'stable/testing/stable' packages (and proper names for those) and a  
few tools to make the management of the repositories an autonomous process 
that get external triggers for decision making.

Some of the ideas are documented in subversion, but again, nothing is set 
into stone.
