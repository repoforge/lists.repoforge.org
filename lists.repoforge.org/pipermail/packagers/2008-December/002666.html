> On Mon, 29 Dec 2008, Christoph Maser wrote:
>
> > i have commited a new php-rrdtool package. It compiles fine on c5 and
> > fc10. on c4 there are some autoconf issues. Since upstream does not have
> > a version i used the release data as Version.
>
> I modified the SPEC file to follow the same structure as all the other
> SPEC files and fixes some problems, the changes should all be trivial.
>
> However using a timestamp can be very problematic in the future when
> someone decides that using real versions makes more sense (or when the
> bindings become part of rrdtool at last).
>
> So it is much better to use 0.0.<timestampe> so we can still move to
> 1.2.30 if that is needed.
>
Well the version looks really ugly to me and i don't think the merge
will ever happen, but of course you are right there.
>
> > I wonder if the package should require the exact rrdtool version it was
> > compiled with?
>
> Good question. In a lot of cases we don't do it because it requires a
> specific ABI version, and often either the soname does not reflect an ABI
> version _and_ because sonames in itself seldom change when the ABI
> changes.
>
> So in those cases it is better to check for yourself for
> incompatibilities, instead of being forced to push a new php-rrdtool
> because you updated the rrdtool package.

So for each rebuild of rrdtool check if the old php-rrdtool package
still works with the new rrdtool package? How do i do that? I don't have
any test-code for php-rrdtool.

> Another option is to make the php-rrdtool part of the rrdtool package (and
> only enable it for RHEL5 since that one builds).

That is a possibility, but it's not my favourite. It will bloat the
rrdtool spec. Also php-rrdtool is some (untested?) contrib software, so
it does not really belong there.
