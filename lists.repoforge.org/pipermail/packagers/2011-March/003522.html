On Thu, 24 Mar 2011, Yury V. Zaytsev wrote:

> Just wanted to ask how complicated it would be to hook dar up with a few
> lines of code to post a tweet whenever you build something and it
> finishes successfully or fails?
>
> Fabian has this kind of thing for his PPC builds and I hope he can share
> his code with you...
>
> It doesn't have to be Twitter actually, maybe even better to have it as
> an RSS feed (I hope tweets can be exported as RSS, so that I can have
> them together with my regular mail) or a static page with the results
> for latest 100 builds or so with successful builds highlighted with
> green and failed with red.
>
> The thing is that you trigger the builds when you have time and there is
> no way to predict when this happens. Sometimes I happen to check if the
> packages that I submitted build fine, but most of the times users detect
> build failures because the packages do not install ages later.
>
> If we would have something like that it would be so much easier to check
> if the things got built fine with dar and act upon if necessary...

Sounds good, the only issues I see are:

  - If something fails I look at the buildlogs and re-issue a build, so the
    twitter wil have much more build-problems than in reality. It would be
    nicer if a successful build negates the previous failed build.

  - A lot of builds are never expected to succeed (eg. dia on RHEL2), and
    these may give a wrong indication of what needs to be looked at.

  - It shouldn't slow down the individual builds (eg. when there are
    network issues)

I already have a log with all successes and failures, I could share that 
one somewhere, but it suffers from the same issues (and it is not being 
rotated either).

I also have a cronjob running with a repoclosure script, although it 
still reports too many false negatives. Need to look at that again too :-/

Personally I would prefer proper infrastructure that more people could 
collaborate with...
