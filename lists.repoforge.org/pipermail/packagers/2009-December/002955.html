On Tue, 15 Dec 2009, Yury V. Zaytsev wrote:

> On Tue, 2009-12-15 at 15:04 +0100, Dag Wieers wrote:
>
>>> Well, actually, I think all we need is just another dar tag, say "risky"
>>> so that all that is tagged as risky goes to a separate repo.
>>
>> That certainly is a possibility. The problem however is interdependencies
>> between the non-risky and risky repository.
>
> Christoph: +1.
>
> I think that on one hand we can freely make having the non-risky
> repository installed a pre-requisite for using risky stuff, but on the
> other hand, the risky packages should not be required for the non-risky
> repo.
>
> So I don't really see a problem here.

The problem starts when you have perl modules in risky that suddenly are 
required by a package that used to be in non-risky. Do you move the 
package to risky ? What happens with people that have it installed but 
don't want all the other packages in risky ?

First they have to know this happens, second they will have to enable 
risky and then include the one that is needed for this package. Thirdly, 
whenever the package dependencies change, and more perl modules from risky 
are needed, they get dependency errors from yum and will have to allow 
more packages from risky.

This to me is not a pleasant or acceptable situation.


>> What really is needed is allow yum to understand that it only needs to
>> update a package coming from the same repository than the one that is
>> installed (unless forced).
>
> OK, I hear you. I see two possible solutions then.
>
> 1) "Fix" yum by writing a plugin that does this and require it by
> rpmforge-release.
>
> I'm not knowledgeable enough nor have enough time to do this, but if we
> shout about that on the list and really consider it a solution that is
> worth to be pursued / document it on the wiki, maybe someone can finally
> come and tackle this?

I have been waiting for that a few years now :-)


> 2) "Fix" RPMForge by introducing two separate repositories (+ keep the
> test pool for experiments).
>
> What do you think about this? Is it more or less practical then "fixing"
> yum? Again, if this is well-documented, we adopt a strict policy about
> that and dar/builders are updated we can slowly clean up this mess.

It's easy to do, we have discussed this in the past. But there is a lot of 
complexity that makes this something to avoid, in my opinion. One such 
complexity is the fact that we build for more than one distribution and 
there are per-distribution differences.
