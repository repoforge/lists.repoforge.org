On Mon, 2010-08-23 at 09:54 +0200, Fabian Arrotin wrote:

> - Problem to be solved : browsing the actual rpmforge/rpms tree through 
> the Trac soucebrowser really kills the machines (both the server and the 
> client side where firefox just hangs for a while). I don't know what can 
> be done to boost perfs or use caching but we started to investigate.

There's nothing that you can do. 

The reason why it's so slow is that there are around 5000 packages in
the repository and it's even slow to enter this directory (before the
caches get populated), not even speaking about downloading a generated
web page of this size.

To make things worse, clever Trac's repository browser actually queries
a lot more than just a list of folders to be able to show fancy
fold-unfold effects.

The only solution you can come up with is to break the list into pages.
There is a ticket for this, but unfortunately, it hasn't been
implemented yet:

http://trac.edgewall.org/ticket/6101

So right now, the only solution is just to redirect everybody to ViewVC
and disable internal repository browser for the time being.

In what concerns other repository browsers, as Karanbir pointed out,
some of the repository browsers for git implement both paging inside the
project and in the project list.

Regarding gitweb specifically, there's a patch for project pagination
which will hopefully make it into git soon:

http://www.spinics.net/lists/git/msg138654.html

So I would suggest not to worry about it at the time being. 

ViewVC solves the problem right now and by the time we migrate to
something else, either Trac will finally support pagination or the
repository browser for whatever new system it will be.
