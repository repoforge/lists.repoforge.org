> Great post Steve. Thanks for taking the time to discuss this topic so
>  detailed. Another point is that rpm-macros-rpmforge is not only
> needed for perl packages. Just have a look at it 
> http://svn.rpmforge.net/svn/trunk/rpms/rpm-macros-rpmforge/rpm-macros-rpmforge.spec
> it disables a lot of things for a lot of distros (yes outdated ones,
> this package didn't get a lot of love lately ;)
> 
I'm not sure those feature macros are actually used at the moment.

If you look at packages like graphviz and mplayer you'll see all those 
feature macros defined for each supported release right inside the specfile.
That has been the style used for years and I gather this was done to 
make the specfile independent of rpm-macros-rpmforge and the rpmforge 
buildsystem.
Until you added the filter* macros to rpm-macros-rpmforge it was never 
necessary to have it in installed when building rpmforge specfiles.

I'm not against having rpm-macros-rpmforge become a requirement for 
building rpmforge packages. It would be nice to not have to duplicate 
those feature macros inside the specfiles.
Also Dries started looking into this for rpmrepo before it stagnated.
(see http://rpmrepo.org/pipermail/rpmrepo-devel/2008-April/000413.html)
