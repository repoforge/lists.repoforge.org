On Thu, 17 Feb 2011, Denis Fateyev wrote:

> Now I look into the "hdapsd" package spec and see a bit strange (well, at my
> opinion) things inside:
>
> ---------
> Name: hdapsd
> %define real_version 20070524
> Version: 0.0
> Release: 0.20070524%{?dist}
> ---------
>
> What was the reason to introduce version as '0.0', separate it with the real
> version (20070524) and put it in the "release" field, next with the "dist"
> tag?
>
> Probably, it's better to do as EPEL packagers did with ntfs-3g (the same
> version naming method, which is popular in recent time) and just remove this
> "real_version" definition entirely:
>
> ---------
> Name:        ntfs-3g
> Version:    2011.1.15
> Release:    1%{?dist}
> ---------
>
> Any thoughts?

We used to use 0.0 versions for anything that was not reasonable. The main 
reason is that some projects use timestamps during initial development and 
then at some later point decide to switch to proper versioning as soon as 
they finish their first 'stable' release.

Going from 20110217 to 1.0 is a major problem that can only be resolved by 
introducing an epoch, which is evil ! It happened before.

So that's why. Nowadays we fall back to what other projects use and hope 
upstream never releases the same package with a lower epoch.
