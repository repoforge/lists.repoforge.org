On 07/07/2010 13:19, Dag Wieers wrote:
>    We could also decide to use a macro for the repotag so that we can make
>    the repotag conditional (eg. use rfx for el5, but rf for all others)
>    although that complexity may be unwanted ?

I actually prefer this. That way the packager can ignore the target, and
buildsystem / policy enforcement can happen at the repository ( and
therefore automation ) level.

>  - We could do the same as above for testing, and use the 'rft'
>    distribution tag instead of the 'test' tag we currently use ?

This would mean that moving from testing -> stable will need a rebuild
to redo the tag. And that in turn would mean the buildroot might not
look the same anymore ( other packages might have changed etc ); so the
testing result would need to be taken with a pinch of salt. Would this
be a big deal ?

Also, i think the 'test' tag is clearer in scope.
