On Jul 7, 2010, at 8:19 AM, Dag Wieers wrote:

> - A new repository tag, eg. 'rfx' for packages that replace upstream
>   packages. Based on this tag, which is set by the Comment-header
>   '# Tag: rfx' we could have these packages show up in the RPMforge
>   Extras repository. A better name may be required, but keep it simple :)
> 
>   We could also decide to use a macro for the repotag so that we can make
>   the repotag conditional (eg. use rfx for el5, but rf for all others)
>   although that complexity may be unwanted ?
> 
> - We could do the same as above for testing, and use the 'rft'
>   distribution tag instead of the 'test' tag we currently use ?

i like both these ideas; however, i'd recommend implementing the metadata differently.  instead of 

# Tag: rfx

i'd recommend

# ReplaceBase: el4 el5

with syntax similar to our existing ExclusiveDist tag.  i can certainly imagine a package which would clobber a base package in one dist but not in another (i don't think it's likely that we'll see similar behavior with regard to arch).

therefore if foo.spec contains the metadata

# ReplaceBase: el5

it would build the packages

foo-x.y-z.el4.rf.noarch.rpm
foo-x.y-z.el5.rfx.noarch.rpm

> - I would also like to change the default buildroot for all SPEC files
>   to:
> 
> 	%(mktemp -ud %{_tmppath}/%{name}-%{version}-%{release}-%{dist}-%{arch}-XXXXXX)
> 
>   which would already be very useful to me, but could be equally useful
>   to others.

i have no strong feelings about this one way or the other.  go for it.

-steve

p.s. i would like to propose that instead of "RPMforge Extras", we refer to the new replacing-core-packages repo as "RPMforge eXtreme" :)
