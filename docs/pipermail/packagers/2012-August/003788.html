> What are the right steps to take next here?  What git commands (a)
> update my fork with the latest stuff from
> repoforge/repoforge.github.com and (b) pull all of those files down
> into ~/repoforge.github.com/ ?

The easiest would be to get them in sync again. First, you need to add
the canonical repository as one of the "remotes" you want to work with:

    git remote add repoforge git://github.com/repoforge/repoforge.github.com.git

N.B.: You need to do that only once per repository clone. Git will
remember about the new remote from now on... 

Then you need to download whatever data there is:

    git fetch repoforge

Finally, you need to reset the state of your current branch to the same
state we have in the canonical repository right now: 

    git reset --hard repoforge/master

N.B.: This is normally a very dangerous destructive command. It deletes
all of your local changes to the branch and resets it to the specified
state, there is no merging going on here. You can easily lose your work
if you do that mindlessly... 

Now you can push this to your fork and appreciate how they are now
perfectly in sync: 

    git push -f origin

N.B.: You need the -f switch, because you are destroying your previous
work pushed to the fork. By default, git would not allow you to do that.
However, here, this is exactly what you want to achieve... 

>From now on, you can again do some work on your checkout, commit it,
push it to your fork and submit pull requests.

The downside is, however, that you can only handle one pull request at a
time this way. If you want to be working on several problems
simultaneously, then you need to learn how to branch properly.

Also, if somebody has changed the canonical repository between you
resetting your fork and submitting pull request, I will have to rebase
your changes on top of that, but this is also an advanced topic.
